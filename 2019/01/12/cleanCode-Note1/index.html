代码整洁之道 Notes-1

2019-01-12

Contents

  1. 🐷命名
  2. 🐷函数
    1. 🐷函数参数

寒假阅读的第一本书是《代码整洁之道》，回想起自己以前写代码的时候犯下的一些错误，给自己后面带来的麻烦，书中都有提到，读来确实感觉有很多收获。

在笔记中也主要是记录自己没有注意的内容，将一些可能很重要但是平时已经做到的内容省略。

🐷命名

  • 避免误导：编程语言中可能有很多专有类型，例如 List, 如果某个变量不是这种类型，就不要将这个关键字用于命名，例如 accountList, 可以用 accountGroup 等代替
  • 尽量用长名字描述清楚变量的含义，不要使用简短的名字或是缩写
  • 将各种常量都定义为可以搜索的名称，这将便于搜索和后续的修改
  • 将一整个工程/模块中的同一个概念对应同一个词，例如以下三者不要混用
    1. controller
    2. driver
    3. manager

🐷函数

函数的第一规则是要短小，第二条规则是还要更短小。 函数应该只做一件事。

  • 尽最大可能地拆分函数。
  • if-else 语句和 while 语句中的代码块应当尽量短小，可以拆分另一个函数出来用于专门处理这种情况，尽量避免多重嵌套
  • 对“函数应该只做一件事”的详细解释：做的事情在同一抽象层面上

  • 函数的整体组织形式应当可以用 “To” 语言来清晰地，自顶向下地描述：
    1. To do A, firstly do B, if C, then do D
    2. To do B, firstly do B1, then do B2
    3. To judge C, firstly do E, then judge F
    4. To do D, firstly do D1, then do D2
  • 注意每一行 To 仅仅只能对应 1 个函数，并且这行描述应当是简洁而清晰的。如果过长则说明应当拆分函数。

🐷函数参数

  • 函数参数地数量应当越少越好，如果超过了 3 个，则必须进行封装，将逻辑上有关联的几个变量封装成一个 class
  • 不要使用“输出参数”，即函数返回值为 void, 而对于其中某一个参数进行转换。这不直观，也不符合正常人的思维方式
  • 当一个函数要对某些实体的状态作出修改时，只能够修改其所属的对象的状态

  • 绝对不使用标识函数（某个输入变量为 boolean, True 代表做某件事，而 False 代表做另一件事）。这直接违反了一个函数只做一件事的原则，应当将其拆分为 doSomeThingForCaseA(), doSomeThingForCaseB() 两个函数
  • 如果一个函数有两个参数，那么两者的位置就可能在使用时搞反。虽然现在的 editor 或者 IDE 可能有提示的功能，但是有时候提示确实没有那么好用。

  • 函数要么做什么事，要么回答什么事，两者不可兼得。如果函数做什么事情之后可能出错，直接 raise Exception, 而不是通过返回错误码
