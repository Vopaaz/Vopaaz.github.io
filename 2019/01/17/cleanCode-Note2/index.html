注释

好的注释也无法弥补烂的代码，因此应当尽可能让代码能够自己说明自己（通过良好的命名和足够细的拆分）。

好的注释可以有以下几种：

  • 简短的法律信息
  • 提供基本信息
  • 解释意图
  • 阐释特别复杂无法拆分的函数，或返回值的意义
  • 警示一个乍看起来并不合理的操作为何必须如此
  • TODO

其他的注释都是多余的！减少不必要的注释也是代码整洁的重要部分。【因此毛老师的 Java 课的注释其实确实没什么必要好写的😀

格式

纵向

单个源代码文件的长度：对于 Java, 尽量限制在 500 行以内。那么对于 Python 而言这个数字应当可以进一步减小，尽量控制在 300 行以内。

关系密切的概念应当相互靠近

  • 变量的声明应当尽可能靠近其使用位置
  • 循环中的控制变量总是在循环语句中声明
  • 将相关的函数尽可能放在一起。按照调用者在上，被调用者在下的顺序紧密排列。这一点可以参考 Notes 1 中提到的语句 “To” 描述规则，两者是相符的。

横向

一行代码必须有长度的限制。这一点可以通过 editor 的 auto format 工具完成，无需特别注意。

协作

团队合作中必须要有一个一致的规则。

对象和数据结构

尽可能将这两种概念区分开：

  • 对象：不直接提供数据，而提供操作数据的函数
  • 数据结构：直接提供数据，不提供函数

它们在定义上应该是完全对立的。

  • 面向对象代码便于在不改动既有函数的前提下添加新的类，但它难以添加新的函数，因为这必须修改继承它、调用它的所有类。
  • 过程式代码（使用数据结构的代码）便于在保持原有数据结构不变的情况下添加新函数，但是如果要改变原有的数据结构，则必须修改所有函数

根据场景来选择该使用哪种实现方式。

一些感想

对于这整个章节，我认为应当注意 Java 和 Python 之间的不同。前者的 private 关键字让私有变量的设定变得非常容易，但是 Python 的双前下划线私有变量设置非常不自然，如果应用很多我觉得代码本身就很丑陋了…好像也没有看到什么 Python 项目里出现 setter or getter 这种东西。两者在设计上是有着根本的不同的。

【更正： Python 有 @property, @some_attr.setter 两个装饰器可以起到这个功能】

实际应用的时候可能更需要关注这种思想，而不是书中介绍的在 Java 中的实现。【嗯不用说了，应用思想仍然是一句屁话x】但至少有一点确实可以关注一下，即对于对象，应该要求它做点什么，而不是给出它的内部情形。

这一点在 cnki2bib 中的体现是，BibTeXEntry 有一个方法是 toBibFileString, 也即要求这个对象给出其最终打印的字符串，而不是在外部函数中一个一个取出它的 field 再进行组合。这会导致函数的抽象不在同一个层面上。
